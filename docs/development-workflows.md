# Development Workflows

This repo is a live home directory, so the workflow is more defensive than in a
normal application repo. The goal is to change the intended files without
touching unrelated personal state, generated artifacts, or separate repos.

## First Commands

From `/Users/benbernard`:

```bash
git status --short --branch
git ls-files
```

Use `git status --short --branch` as the default verification anchor before and
after work. If the tree is dirty, treat existing changes as user-owned unless
you made them in the current turn.

## Safe Discovery

Good discovery commands:

```bash
git ls-files 'docs/*'
git ls-files 'bin/ts/src/*'
rg 'meeting-notify' $(git ls-files)
rg --files -g '!repos/**' -g '!**/node_modules/**'
```

Avoid:

```bash
find . -type f
rg pattern .
grep -R pattern .
```

Those commands can walk `repos/`, dependency trees, and submodules. If you need
one of those areas, target it directly and exclude `node_modules/`.

## Editing Rules

Use `apply_patch` for manual file edits. Do not overwrite files with ad hoc shell
redirection when a patch is practical.

Do not revert user-owned changes. If an unrelated file is already modified,
ignore it. If a modified file is part of your task, read it carefully and edit
around the existing change.

Do not edit generated artifacts:

- `bin/ts/bin/`
- `bin/ts/dist/`
- `bin/ts/src/<command>` symlinks generated by the build

For new personal utilities, prefer TypeScript in `bin/ts/src/` over adding a new
mixed-language script in `bin/`.

## Checks

Use checks that match the edited surface.

Markdown or documentation only:

```bash
git diff --check -- docs
```

Shell or dotfile changes:

- Prefer targeted syntax checks when available.
- Do not run `bin/ts` checks unless TypeScript files changed.

`bin/ts` TypeScript changes:

```bash
cd bin/ts
bun run typecheck
bun test
```

For broad TypeScript changes, also run:

```bash
bun run check
```

Note that `bun run build` runs a `prebuild` script with `biome check
--apply-unsafe .`, which can rewrite files. Use it intentionally.

## Secrets And Internal Config

This repo tracks many config files. Never add:

- API keys, tokens, passwords, or cookies.
- Internal URLs or job-specific values unless they are already intentionally
  tracked and necessary.
- Personal identification data beyond what is already present in tracked config.

Be especially cautious in:

- `.config/`
- `.claude/`
- `.gitconfig`
- `site/` references
- generated app config exports

## Source-Control Boundaries

`repos/` is not part of this repo. A task inside `repos/foo` should follow
`repos/foo`'s own git state, tooling, and instructions.

`site/` is a separate repo. It may contain useful overrides, but do not edit it
as part of a home-directory task unless the user explicitly asks.

## Commit And Push Notes

Do not commit unless asked.

When force pushing is necessary, prefer:

```bash
git push --force-with-lease
```

If Codex blocks that command because `CODEX_CI=1` is set, run the command with
`CODEX_CI=0` or `env -u CODEX_CI` and tell the user.

## Root Agent Files Versus Docs

Keep root agent instruction files directive and short. Durable explanations,
architecture, and runbooks should live in `docs/` or in the relevant subsystem
directory, such as `bin/ts/README.md`.

Root `AGENTS.md` should point agents toward docs rather than duplicating every
detail.
